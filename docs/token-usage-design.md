# Token Usage in Atom Agent

## Where do the numbers come from?

Every token count you see in the agent output is **reported by the LLM provider** (Anthropic or OpenAI), not estimated locally. When the agent makes a streaming API call, the provider includes a `usage` object in the final SSE chunk of the response containing the exact token counts as measured by their server-side tokenizer.

Pydantic AI captures this metadata from the raw API response and exposes it through `stream.usage()` (per-turn) and `result.usage()` (accumulated total).

## Reading the usage line

Here's an example of the agent's usage output:

```
📊 [562,729 in (4,854 new · 47,910 cache write · 509,965 cache read) / 46,763 out | 20 reqs / 19 tools]
```

Breaking it down:

| Field | Meaning |
|---|---|
| `562,729 in` | Total input tokens sent to the model |
| `4,854 new` | Input tokens that were neither cached nor written — typically the latest user message after the last cache breakpoint |
| `47,910 cache write` | Input tokens newly stored in the cache on this run (costs 25% more than base input rate, but saves money on subsequent calls) |
| `509,965 cache read` | Input tokens served from the cache (costs only 10% of base input rate) |
| `46,763 out` | Output tokens generated by the model (text responses + tool call arguments). **Output tokens are never cached** — they are always fresh |
| `20 reqs` | Number of LLM API calls made so far in this turn |
| `19 tools` | Total number of tool invocations across all those calls |

The three input categories always add up to the total: `new + cache write + cache read = in` (4,854 + 47,910 + 509,965 = 562,729).

## Why is there no cache breakdown for output tokens?

Caching only applies to **input** tokens. Input tokens are the prompt you send to the model (system prompt, conversation history, tool results) — these are often identical or overlapping across calls, so Anthropic can cache and reuse them. Output tokens are what the model *generates*, and those are unique every time — there's nothing to cache.

That's why `out` has no sub-breakdown: it's always "new."

## Per-turn vs total usage

The agent operates in a loop: think → tool call → observe result → think again → ... → final answer. Each iteration through this loop involves a separate LLM API call, and each call returns its own usage.

```
💬 I'll create a home page for you...
  📊 [1,033 in (0 new · 1,033 cache write · 0 cache read) / 82 out | 1 reqs / 0 tools]
🔧 Tool: list_dir(...)
💬 Let me now write the HTML...
  📊 [13,842 in (0 new · 5,306 cache write · 8,536 cache read) / 6,813 out | 6 reqs / 12 tools]
🔧 Tool: write_file(...)
💬 Here's your home page!
  📊 [47,202 in (4,854 new · 8,536 cache write · 33,812 cache read) / 720 out | 8 reqs / 14 tools]

📊 Total: 47,202 in (4,854 new · 8,536 cache write · 33,812 cache read) / 720 out / 47,922 total | 8 reqs / 14 tools
```

Per-turn counters are **cumulative** — the numbers include all prior calls in the same user turn, not just the latest call.

## Requests and tool call counters

- **`requests`** — the number of LLM API calls made so far in this agent run. Each time the agent sends a prompt to the model and gets a response, this increments. A multi-step agent run (think → tool → think → tool → answer) will make several requests.
- **`tool_calls`** — the total number of tool invocations across all requests. A single LLM response can call multiple tools in parallel (e.g., `list_dir` + `read_file` in one shot), so `tool_calls` can grow faster than `requests`.

## Why input tokens grow each turn

`input_tokens` increases with each turn because the full conversation history (system prompt + all previous messages + tool results) is re-sent to the LLM on every call. The model is stateless — it has no memory between calls — so the context window grows as the conversation progresses.

```
Call #1:  system prompt + user message                              → 1,033 input tokens
Call #2:  system prompt + user message + turn 1 + tool result       → 13,842 input tokens
Call #3:  system prompt + user message + turns 1-2 + tool results   → 47,202 input tokens
```

This is why Anthropic prompt caching is enabled in the agent config:

```python
model_settings={
    "anthropic_cache_instructions": True,
    "anthropic_cache_tool_definitions": True,
    "anthropic_cache_messages": True,
}
```

## How the three input categories work

Anthropic's API returns three input token categories (Pydantic AI field names in parentheses):

- **`cache_read_input_tokens`** (`cache_read_tokens`) — input tokens served from the prompt cache. These are billed at 10% of the base input rate (a 90% discount).
- **`cache_creation_input_tokens`** (`cache_write_tokens`) — input tokens newly written to the cache. These cost 25% more than the base input rate, but subsequent calls that share the same prefix will read from cache instead.
- **`input_tokens`** — input tokens that were neither read from nor written to the cache (i.e., tokens after the last cache breakpoint).

The agent computes `new` as `input_tokens_total - cache_write - cache_read` to show all three categories.

### Why cache read grows over turns

On the first LLM call, nothing is cached — the system prompt, tool definitions, and user message are all new. Anthropic caches this prefix. On the second call, that entire prefix is a cache read, and only the new content (tool results, assistant response) becomes a cache write. By the third or fourth call, the majority of input tokens are cache reads:

```
Call #1:    0 cache read /  1,033 cache write  (everything is new)
Call #2:  8,536 cache read /  5,306 cache write  (system + tools + turn 1 cached)
Call #3: 33,812 cache read /  8,536 cache write  (most of the prefix is cached)
Call #4: 80%+ cache read  /  ~10% cache write   (very little new content)
```

### Cost impact

For a typical multi-step agent run with ~500K input tokens:

| Scenario | Effective input cost |
|---|---|
| No caching | 500K × base rate |
| 90% cache read | 50K × base rate + 450K × 0.1 base rate = ~95K equivalent |

Caching can reduce input token costs by **50–80%** on multi-turn agentic workloads.

## What is a token?

A token is not a character or a word — it's a subword unit determined by the model's tokenizer. Some rough rules of thumb:

- English prose: ~1 token per 3–4 characters
- Code and JSON: token density varies widely (whitespace, brackets, and quotes can each be separate tokens)
- Non-ASCII text: often more tokens per character

For example, the string `"Hello, world!"` is 4 tokens in Claude's tokenizer, while `{"path": "/workspace/index.html"}` might be 12+ tokens.

## Cost implications

Anthropic charges per token (rates vary by model). The per-turn usage lines let you see exactly where tokens are being spent:

- **High `new` input tokens**: the prompt has a lot of uncacheable content after the last cache breakpoint.
- **High `cache write`**: new content is being written to the cache — expected on early calls, costly if it happens repeatedly (cache may be expiring).
- **Low `cache read`**: the cache isn't being hit effectively — check that the prompt prefix is stable between calls.
- **High output tokens**: the model is generating large file contents as tool arguments (e.g., writing a full HTML file via `write_file`). This is expected for code-generation tasks. Output is never cached.
- **Many turns with growing input**: each agentic loop iteration re-sends everything. Prompt caching helps, but very long multi-step tasks will accumulate significant input token costs.
