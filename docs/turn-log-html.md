# Turn Log to HTML Converter

Convert turn logs (MIME multipart format) generated by `TurnLogger` into a beautiful, interactive HTML report with a clean table layout.

**Script:** `agent/turn_log_to_html.py`

## Features

тЪЫя╕П **Table Layout** тАФ Clean, scannable table with # | Q/T/S | TYPE | CONTENT columns  
ЁЯФН **Inline Search** тАФ Search box on same line as CONTENT header, filters all columns with yellow highlighting (max 100)  
ЁЯОп **Clickable TYPE Column** тАФ Click any TYPE badge to filter; popup appears in top-right for multi-select  
ЁЯУМ **Sticky Header** тАФ Table header stays visible when scrolling through long logs  
ЁЯОи **Brand Colors** тАФ Uses #0053e2 (blue) and #ffc220 (spark yellow)  
ЁЯУЭ **JSON Prettifier** тАФ Click 'pretty' to format JSON, 'plain' to collapse (EXEC only)  
ЁЯУК **Compact Design** тАФ Minimal padding, maximized content space  
ЁЯУН **All Visible** тАФ No collapsing, everything shown by default  
ЁЯФО **Grep-Friendly** тАФ Source logs remain plain text for debugging

## Usage

### Basic

```bash
python agent/turn_log_to_html.py ~/atom-agentic-ai/logs/session-abc-123/
```

This generates `session-abc-123.html` in the parent directory.

### Custom Output

```bash
python agent/turn_log_to_html.py ~/atom-agentic-ai/logs/session-abc-123/ -o my_report.html
```

### Auto-Open in Browser

```bash
python agent/turn_log_to_html.py ~/atom-agentic-ai/logs/session-abc-123/ --open
```

## Turn Log Format

Turn logs are created by `agent/turn_logger.py` and follow this naming convention:

```
q{QQ}.t{TT}.s{SS}.{type}.{label}.txt
```

Where:
- `QQ` = Query number (user prompt)
- `TT` = Turn number (model request within a query)
- `SS` = Sequence number (logged item within a turn)
- `type` = `thinking` | `text` | `plan` | `exec`
- `label` = 50-char description (auto-sanitized)

Each file contains MIME multipart content:

```
Boundary: ----=_Part_abc123...
Timestamp: 2026-04-09T20:26:27.370560+00:00
Query: 1
Turn: 1
Tool: execute_command
Call-ID: toolu_vrtx_015vZN58xeLVH27y1X9RPwgv

----=_Part_abc123...
Content-Type: input

{"command": "ls -la", "working_dir": "."}

----=_Part_abc123...
Content-Type: output

Exit Code: 0
STDOUT:
total 48
drwxr-xr-x  12 user  staff   384 Apr  9 20:26 .
...

----=_Part_abc123...--
```

## HTML Output

The generated HTML report displays entries in a compact table format:

### Table Columns

| Column | Width | Alignment | Description |
|--------|-------|-----------|-------------|
| **#** | 25px | Right | Line number (minimal padding) |
| **Q/T/S** | 55px | Right (header centered) | Query/Turn/Sequence coordinates |
| **TYPE** | 60px | Left | Color-coded entry type badge |
| **CONTENT** | Remaining | Left | Full content with auto-wrap |

### Entry Type Display

**TEXT entries** тАФ Just the badge, clean content:
```
TEXT
I'll help you find when the `phoa` value is NOT set to `subAppId`...
```

**PLAN entries** тАФ Badge + tool name + raw content (no prettify button):
```
PLAN
agent_run_shell_command
{"command": "git clone git@github.com:user/repo.git", "working_dir": "."}
```

**EXEC entries** тАФ Badge + tool name (once) + labeled input/output sections with prettify:
```
EXEC

agent_run_shell_command | input ЁЯкД
{"command": "git clone...", "working_dir": "."}

output ЁЯкД
Exit Code: 0
STDOUT:
Cloning into 'my-server'...
```

### Color-Coded Type Badges

- ЁЯТн **THINKING** (purple) тАФ Extended reasoning
- ЁЯдЦ **TEXT** (blue) тАФ LLM text responses
- ЁЯУЛ **PLAN** (yellow) тАФ Tool calls LLM wants to make
- тЪб **EXEC** (green) тАФ Tool execution results

### Interactive Filters

**Inline Search:**
- Search box located on the same line as "CONTENT" in the column header
- Type any text to filter rows across ALL columns (not just CONTENT)
- Case-insensitive search
- Searches Q/T/S, TYPE, and CONTENT columns
- Real-time filtering as you type
- **Matched text is highlighted in bright yellow** (#ffc220) for easy scanning
- **Smart highlighting limits**: Highlights first 100 matches for performance
- Clear search box to restore all visible rows (respects TYPE filters)

**Clickable TYPE Column & Multi-Select Popup:**
- **Click any TYPE badge** to show ONLY that type + highlights the row
  - A filter popup appears in the **top-right corner** with checkboxes for all types
  - Each type shows its count in parentheses: e.g., `ЁЯдЦ TEXT (12)`, `тЪб EXEC (5)`
  - Check/uncheck types to show/hide multiple types at once
  - Click "Show All" button in popup to clear all filters, hide popup, and scroll to show top of highlighted row
- **Click the same TYPE badge again** to:
  - Show all types (hides the popup)
  - Jump to that row (instant, top-aligned - # column visible at top)
  - Keep the row highlighted
- **Persistent highlight:** Row stays highlighted until you click another TYPE badge
- **Smart scroll:** Scrolling always aligns the top of the row with the viewport (# column visible)
- **Hover effect:** Rows get a light gray background when cursor hovers over them
- Hidden types are dimmed (30% opacity)
- Clicking/selecting text in CONTENT column does NOT clear the highlight
- Works in combination with text search

**Entry Types:**
- ЁЯТн **THINKING** тАФ LLM reasoning and thought process
- ЁЯдЦ **TEXT** тАФ LLM text responses
- ЁЯУЛ **PLAN** тАФ Tool call plans (what LLM wants to execute)
- тЪб **EXEC** тАФ Tool execution results
- тЪЩя╕П **META** тАФ Session metadata (model, MCP config, available tools)
- ЁЯУЬ **SYS** тАФ System prompt for each query
- ЁЯСд **USER** тАФ User input prompts

**Perfect for:**
- Focusing on specific conversation aspects:
  - Click ЁЯдЦ TEXT to see only LLM responses
  - Click тЪб EXEC to see only tool execution results
  - Click ЁЯТн THINKING to see only reasoning process
  - Click ЁЯСд USER to see only user inputs
- Finding specific tool calls or error messages with instant visual feedback
- Debugging multi-turn flows by searching for specific values
- Performance optimized: limits highlighting to first 100 matches on large result sets

### Design Philosophy

- **Minimal padding** тАФ First 3 columns use `py-1` and `pr-1` only
- **Fixed widths** тАФ Metadata columns as narrow as possible
- **Sticky table header** тАФ Column headers remain visible when scrolling
- **Inline search** тАФ Search box in CONTENT header, filters all columns with clickable TYPE badges
- **Auto-wrap content** тАФ Text wraps in CONTENT column for readability
- **All visible** тАФ No collapsing, full content displayed immediately
- **Consistent spacing** тАФ Both badges and content boxes use `px-2 py-0.5`
- **Tool names** тАФ Shows which tool is being called for PLAN and EXEC entries

## Why HTML?

Turn logs are great for grep/debugging, but when you need to:
- Review a long conversation flow (236+ entries!)
- Share with non-technical stakeholders
- Understand tool execution sequences
- Debug multi-turn agent loops
- See everything at once without clicking

...a clean, table-based HTML view beats navigating 200+ plain text files! ЁЯР╢
