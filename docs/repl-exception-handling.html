⚛️ Atom Agent — REPL Exception Handling

How repl.py recovers from cancellations, API timeouts, and network failures without crashing the interactive session.

1 Design Principle

Two-Layer Architecture
INNER (inside async with agent.iter()) — sets flags, runs finally: _finalize_turn(), then re-raises. No logging, no user messages.
OUTER (await task boundary) — single authoritative handler. Logs, prints user messages, and decides swallow vs. re-raise.

The inner layer must exist because _finalize_turn(run, …) needs the run object alive — it can only be called inside async with agent.iter() as run. But it does zero decision-making; it just tags the exception type onto flags and lets Python's natural propagation carry the exception to the single outer handler.

3 Exception Types & Outcomes

Exception Source Flag Outcome
asyncio.CancelledError Ctrl+C → SIGINT → task.cancel() cancelled = True swallowed → REPL
anyio.ClosedResourceError Stream torn down during cancellation cancelled = True swallowed → REPL
httpx.ReadTimeout Anthropic API slow response transport_failed = True swallowed → REPL
httpcore.ReadTimeout Low-level socket timeout transport_failed = True swallowed → REPL
anthropic.APITimeoutError Anthropic SDK wrapping httpx timeout transport_failed = True swallowed → REPL
anthropic.APIConnectionError Network unreachable / DNS failure transport_failed = True swallowed → REPL
ModelAPIError wrapping any of the above pydantic_ai re-wrapping transport errors transport_failed = True swallowed → REPL
httpx.RemoteProtocolError Malformed HTTP response mid-stream transport_failed = True swallowed → REPL
Any other Exception Actual bug in agent code — re-raised → crash

4 __cause__ Chain Walking

Libraries like pydantic_ai and anthropic wrap low-level transport errors in higher-level exceptions via raise X from Y. The outer handler only sees the outermost wrapper. _is_stream_connection_error() walks the full __cause__ chain to detect the original recoverable error at any depth.

Example: The actual exception chain from the bug that motivated this design —

ModelAPIError pydantic_ai.exceptions
→
APITimeoutError anthropic
→
ReadTimeout ✓ httpx
→
ReadTimeout ✓ httpcore

The walker finds anthropic.APITimeoutError at depth 1 and returns True. Without chain-walking, only ModelAPIError (depth 0) would be checked — and it's not in the recoverable set, so the agent would crash. 

def _is_stream_connection_error(exc):
    current = exc
    while current is not None:
        if isinstance(current, _RECOVERABLE_TRANSPORT_ERRORS):
            return True
        current = getattr(current, "__cause__", None)
    return False

5 _finalize_turn() — The Safety Net

Called in finally: on every code path — normal completion, cancellation, and transport failure. It has its own internal try/except to guarantee best-effort partial state preservation even when things go sideways.

_finalize_turn(run, message_history, …, cancelled)
Happy path — run.result exists:
  • Extend message_history with new messages
  • Print agent output (unless cancelled)
  • Accumulate usage stats
  • Log to GCS

Fallback — run.result unavailable (e.g. mid-stream crash):
  • _save_partial_history() — salvage whatever messages exist
  • _sanitize_history() — fix orphaned tool_result blocks
  • _log_partial_usage() — best-effort usage tracking
  • Each of these has its own try/except — they never raise

cancelled=True. Calls _save_partial_history(), _log_partial_usage().
Salvages whatever messages were received before cancellation.
Outer — except (CancelledError, ClosedResourceError)
cancelled = True. logger.info("Turn cancelled by user").

Prints ⚠️ Cancelled.

⏱️ Direct API Timeout

🧅 Wrapped Timeout — ModelAPIError → APITimeoutError → ReadTimeout

Inner — except Exception

  depth 0: ModelAPIError — not in tuple
  depth 1: e.__cause__ = APITimeoutError — match! ✓

raise.
Finally → _finalize_turn()
transport_failed=True. Saves partial history and usage via fallback path.
Outer — except Exception
_is_stream_connection_error(e) → walks chain again → True.
Logs warning. Prints stream-connection-error message.

Prints ⚠️ Turn ended early…
Session saved. Returns to REPL.

🐛 Actual Bug — KeyError in streaming loop

A genuine programming error — e.g. KeyError or AttributeError in tool handling code.

Inner — except Exception
_is_stream_connection_error(KeyError(…)) → walks chain → False.
No flag set. transport_failed stays False.
raise — re-raises the KeyError.
Finally → _finalize_turn()
cancelled=False, transport_failed=False. Best-effort partial history save.
Outer — except Exception
_is_stream_connection_error(e) → False.
Hits the else: raise branch.
Exception re-raised — propagates up to asyncio.run() → full traceback.
Crash
finally: loop.remove_signal_handler(SIGINT) still runs (cleanup).
Traceback printed. Process exits.
💥 Crash with full traceback. This is intentional — real bugs should not be silently swallowed.
